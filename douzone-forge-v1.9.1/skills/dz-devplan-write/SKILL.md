# 개발 계획 수립 작성 스킬 (dz-devplan-write)

> 용어: 개발 계획 수립(코딩 전 무엇을·순서·공수·일정을 정하는 단계) · ATLAS(개발 계획 작성 전담 서브에이전트) · AEGIS(정합성 검증 전담 에이전트, 읽기·검색만) · 공수(일을 끝내는 데 드는 사람·시간, 단위 인일) · PERT(낙관·보통·비관 3점을 가중평균해 기대 공수를 내는 기법) · CPM(임계 경로법 — 의존 관계로 최단 완료 기간 계산) · 보정계수(과거 추정 대비 실측 비율로 차기 추정을 교정하는 배수) · WBS(작업 분해 일정표) · 비강제(opt-in, 작성 여부는 선택)
>
> **SSoT(단일 출처)**: [`규칙/프로세스/개발-계획-수립-표준.md`](../../../../../Workspace/douzone-forge/규칙/프로세스/개발-계획-수립-표준.md)(D-06). 본 스킬은 그 표준을 **실행**하는 절차다. 작성 주체 = ATLAS(`.claude/agents/dz-atlas.md`), 검증 = AEGIS.

## 0. 한 줄 정의

프로젝트(PRJ)·JIRA 요구사항·기획/설계 산출물을 받아 **개발 계획**(① 개발 관점 분석 · ② Task 분해·의존성 · ③ 공수 측정 · ④ 일정·마일스톤)을 D-06 표준대로 **한 파일**로 생성한다. 그 계획에 따라 개발설계(D-08)·코딩이 무엇을·순서·공수·일정 기준으로 진행된다.

## 1. 발동 / 비강제

- 사용자가 "개발 계획 세워줘"·"{모듈} {기능} 개발 계획"·"Task 분해·공수·일정"을 요청할 때.
- **비강제(opt-in)**: 작성 여부는 본인 선택, 기존 진행 건 소급 의무 없음. 이미 PRJ TASK·WBS로 운영 중이면 그 Task를 D-06 필드(produces·depends_on·O·M·P)로 **보강**(중복 생성 금지). **단 작성하면 D-06 정합 형식은 MUST**(작성 여부 ≠ 작성 형식).

## 2. 작성 전 체크리스트

1. **선행 입력 존재** — 프로젝트 정보(PRJ 범위·기간) · JIRA 요구사항 · 요구사항(D-05 AC/REQ) · 화면설계(D-07 SCR) · IA(화면 목록·라이선스 분기). 없으면 먼저 확보(CIPHER·VECTOR·MATRIX 산출).
2. **기존 계획 자산 확인** — 해당 PRJ `02. TASK 목록`·WBS(Google Sheets)에 이미 Task가 있으면 신규 생성 대신 **보강** 대상으로.
3. **실측 누적 확인** — `Amaranth10/{모듈}/tasks/_공수실측.md` 존재 시 보정계수 `k` 입력(없으면 k=1).
4. **대상 모듈 맥락** — 협업계(neos)/ERP계(erp)·라이선스 분기·멀티테넌트 영향 식별(개발 관점 분석 입력).
5. **세션 체크** — `_개인/sessions/{모듈}/_current.md` 진행 중 건이 있으면 이어받기.
6. **투입 인원·Cell리더·가용성 확인(협업 기본)** — 개발 계획은 보통 여러 개발자 동시 투입 + Cell리더 참여. 투입 개발자 목록·Task별 담당자 배정·O·M·P 합의는 Cell리더+개발자 협업으로 진행(단독 추정 지양). **각 투입 개발자의 전 프로젝트 task 부하(가용성)**를 확인(타 PRJ 점유 시 가용 시점부터) — 모든 PRJ TASK·WBS 기준.

## 3. 절차

```
① 입력 수집·경계 확정
   - PRJ·JIRA·요구사항(D-05)·화면설계(D-07)·IA + 모듈 협업계/ERP·라이선스 분기 + 기존 PRJ TASK/WBS + 실측 누적(k)
② ATLAS 에이전트 디스패치 (작성)
   - ③ 개발 관점 분석: 영향 범위·기술 제약·재사용/신규·리스크 (모든 REQ/AC·SCR 매핑)
   - ④ Task 분해 + 의존성: 각 Task에 produces·traces·depends_on. 순환 금지(위상 정렬). 개발설계→구현→단위시험 선후
   - ⑤ 공수 측정: Task별 O·M·P → TE=(O+4M+P)/6 · σ=(P−O)/6. 보정 TE'=TE×k (k 출처 명시, 없으면 k=1)
   - ⑥ 일정·마일스톤: CPM(위상정렬→ES/EF forward→LS/LF backward→slack). 임계경로=slack 0 사슬. 마일스톤 4단계 게이트(M1 개발설계 확정·M3 단위시험·develop). 신뢰상한 ≈ 임계경로 + 1.645×sqrt(Σσ²|임계경로)
   - ⑥' 자원 배치 일정(다중 개발자 + 가용성 — 협업 기본): **개발자 가용성 선결**(전 프로젝트 task 부하 → 가용 시점) → Cell리더가 투입 개발자·Task별 assignee 배분 → 같은 담당자 직렬·다른 담당자 의존성 범위 병렬 → 자원가능시작=max(선행 EF, 같은 담당자 직전 EF, available_from) → 달력 일정(1명=Σ TE' 직렬, N명 ≥ 임계경로). 공수(인일)≠일정(달력), 투입 인원·배치·가용성 가정 명시
   - 추적 키 ①②③ 부착(④ produces·⑤ 구현매핑[후행]) · 미확정 _(확인 필요)_ · 자동화 형식(D-06 부록 B 스키마)
③ AEGIS 검증 (정합성, 읽기·검색만)
   - 추적 키 grep 조인: REQ/AC·SCR이 Task로 분해 안 됐으면 '계획 누락', produces↔D-08 항목 불일치면 '모순'
   - 품질 게이트(D-06 §9): 순환 의존 0 · TE·σ 공식 일치(임의값 0) · 임계경로 slack 0 · 마일스톤 게이트 정렬 · 불확실성 폭 표기
④ WBS 제안 → 확정 (실무 개발담당자/리더)
   - 산출 일정표(WBS)는 **제안(draft)**으로 제시 → **실무 개발담당자 또는 리더(Cell리더) 의견 수렴 → 확정**(자동 산정 ≠ 자동 확정, status 제안/확정 표기)
   - 확정 Task만 개발설계(D-08)·코딩 착수 입력 (미확정·미확정 추측 착수 차단)
⑤ 연계 — PRJ TASK·WBS 정합(보강) + 확정 Task 순서가 D-08(개발설계) 작성 순서. M1이 BLUEPRINT 진입점. 완료 시 actual을 _공수실측.md에 누적(차기 보정)
```

> **자동 산정 핵심**: ⑤공수·⑥일정은 **직관이 아니라 D-06 부록 B 공식**으로 계산한다 — 같은 입력이면 사람·자동화 체계가 같은 결과를 낸다. ATLAS는 Task 입력(O·M·P·depends_on)에서 TE·σ·임계경로·마일스톤을 결정적으로 산출한다.

## 4. 출력

| 산출 | 위치 |
|---|---|
| 개발 계획 (md 1파일, §3~§6 순서) | `Amaranth10/{모듈}/history/plans/{모듈}-{기능slug}-개발계획.md` |
| 실측 누적(완료 후) | `Amaranth10/{모듈}/tasks/_공수실측.md` |

작업 중엔 작업 폴더, 확정 시 위 경로로 편입(파일 명명 CLAUDE.md 규칙 정합). 기존 PRJ TASK/WBS가 있으면 그쪽을 정본으로 두고 본 파일은 개발 관점 분석·공수·일정 상세로 보강 연결.

## 5. 디스패치 예시

> 사용자: "법무 송무 사건비용 개발 계획 세워줘"

→ ① 화면설계(LTEWI019)·요구사항·PRJ·기존 TASK 수집 → ② ATLAS 디스패치(개발 관점 분석→Task 분해→공수 PERT→일정 CPM) → `Amaranth10/법무관리(LTE)/history/plans/lte-사건비용-개발계획.md` → ③ AEGIS 검증 → ④ 사용자 확정 → ⑤ 확정 Task → BLUEPRINT(개발설계 D-08)로 인계.
