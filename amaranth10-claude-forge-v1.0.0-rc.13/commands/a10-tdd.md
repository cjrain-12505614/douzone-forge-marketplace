# /tdd — TDD 워크플로우

테스트 주도 개발(TDD) 사이클을 안내한다.

## 사용법

```
/tdd {기능 설명}
```

예: `/tdd 벌크 상태 변경 API`

## 워크플로우

### Phase 1: RED (실패하는 테스트 작성)
1. 구현할 기능의 기대 동작을 테스트로 작성
2. 테스트 실행 → **반드시 실패 확인**
3. 실패 메시지가 기대한 것인지 확인

### Phase 2: GREEN (최소한의 코드로 통과)
1. 테스트를 통과시키는 **최소한의** 코드 작성
2. 리팩터링 유혹 참기 — 일단 통과만 시키기
3. 테스트 실행 → **통과 확인**

### Phase 3: REFACTOR (코드 개선)
1. 중복 제거, 네이밍 개선, 구조 정리
2. 테스트 실행 → **여전히 통과 확인**
3. 다음 기능으로 Phase 1 반복

## 테스트 네이밍 규칙

```java
@Test
void should_태스크상태변경_when_권한있는사용자() { ... }

@Test
void should_403반환_when_COMMENTER가벌크수정시도() { ... }
```

## 주의사항

- 테스트 없이 프로덕션 코드를 먼저 작성하지 않는다
- 한 번에 하나의 기능만 구현한다
- 리팩터링 단계에서는 기능을 추가하지 않는다
