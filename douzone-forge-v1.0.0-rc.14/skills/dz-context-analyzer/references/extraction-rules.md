# 컨텍스트 추출 규칙 (Context Extraction Rules)

화면설계서 PDF와 스크린샷 분석 시 반드시 추출해야 할 항목과 파일 배치 기준을 정의한다.

## 반드시 추출할 것 (Must-Have Items)

### 1. 기본 정보 (Identity)
- **모듈명**: PDF 파일명 또는 메뉴 구조에서 추출
- **GNB/LNB 위치**: 메뉴 계층 구조 명확화
- **기능명**: 화면 또는 LNB 제목

### 2. 기능 정의 (Functional Definition)
- **기능 목적**: 이 화면/기능이 해결하는 문제
- **화면 구성**: 검색/필터, 목록, 상세 화면, 팝업 구조
- **필드 정의**: 각 필드명, 타입, 필수 여부, 기본값
- **버튼 동작**: 각 버튼의 기능과 연결 흐름
- **상태값/코드값**: 화면에 표시되는 모든 상태, 코드, 선택지

### 3. 관련 데이터 (Related Data)
- **연관 GNB/LNB**: 이 기능과 연계하는 다른 메뉴
- **주요 테이블**: 이 화면이 다루는 주요 데이터 테이블
- **연계 API**: 외부 연계 정보 (있는 경우)

### 4. 라이선스 단서 (License Indicators)
- 화면의 어느 부분이 라이선스별로 다른가?
- "법무법인용에서는 미제공", "기업법무팀용에서만 제공" 등의 표기 확인

### 5. 설계 변경 이력 (Design History)
- Document History 테이블 (버전, 날짜, 변경 내용, 담당자)
- 버전별 주요 결정 사항

## 파일 배치 기준 (File Placement Rules)

### context/{GNB}/{LNB}.md - 현재 기능 정의
**저장 기준**: 현재 운영 상태의 기능 정의

포함 항목:
- 기능 목적과 사용자 유형
- 화면 구성 (UI 요소, 레이아웃)
- 필드 정의 (현재 구현된 모든 필드)
- 버튼 동작과 연결 흐름
- 상태값/코드값 전체 목록
- 라이선스별 기능 범위 (표로 정리)
- 알려진 제약사항 또는 확인 필요 항목

**제외 항목**: 설계 변경 이유, 버전별 변경 내용, 요청 배경

### history/requests/{날짜}-{기능명}-설계히스토리.md - 설계 변경 이력
**저장 기준**: "왜 이렇게 설계했는가"에 대한 이력

포함 항목:
- Document History 테이블 (버전별 정보)
- 버전별 변경 내용과 변경 이유
- 각 버전에서 영향받은 LNB/필드
- 설계 검토 및 승인 정보 (있는 경우)
- 요청자 및 담당자

**제외 항목**: 현재 기능 상세 (이는 context/ 파일에만 있음)

### history/phase-N.md - 개발 차수별 맥락
**저장 기준**: "이 모듈은 왜 이렇게 구성되었는가"에 대한 거시적 맥락

포함 항목:
- 개발 차수의 목표
- 그 차수에서 새로 추가/변경된 주요 GNB/LNB 목록
- 아키텍처 결정사항 (왜 이 구조인가)
- 기술적 제약사항 (당시 기술, 성능, 호환성)
- 알려진 기술 부채

**제외 항목**: 개별 LNB의 필드 정의, 사소한 UI 변경

### history/plans/{날짜}-{주제}-계획서.md - 개발계획서
**저장 기준**: 향후 개발 또는 고도화의 계획

포함 항목:
- 목표와 배경
- 영향받을 GNB/LNB
- 일정 및 마일스톤
- 위험 요소 및 대응 방안

### history/meetings/{날짜}-{주제}-회의유형.md - 회의록
**저장 기준**: 개발 중 나온 결정사항과 액션아이템

포함 항목:
- 회의 참석자
- 논의된 주요 안건
- 결정사항
- 액션아이템 (담당자, 마감일)

### history/reports/{날짜}-{주제}-보고유형.md - 보고자료
**저장 기준**: 진행 상황 또는 현황 보고

포함 항목:
- 보고 대상 기간
- 완료된 작업
- 진행 중인 작업
- 리스크/이슈

## 불확실성 처리 Rules

### 화면에서 직접 확인되지 않는 항목
형식: `_(확인 필요)_` 태그로 표시

예:
```markdown
- 상담 상태: 신규, 진행중, 대기, 완료 _(설계서에 명시, 화면 이미지 확인 필요)_
- 자동 계산 로직: 비용 = 시간 * 요금 _(설계서 텍스트 기반 추론, 실제 구현 확인 필요)_
```

### 텍스트와 이미지가 다를 경우
**원칙**: 화면 이미지(운영 상태) 기준으로 확정

기록 방식:
```markdown
## 설계서 vs 실제 운영 차이
- 설계서: [기능명] → 실제 운영: [다른 기능명]
- 업데이트 날짜: YYYY-MM-DD (스크린샷 기준)
```

### 라이선스 단서가 없을 경우
형식: `_(라이선스 적용 여부 확인 필요)_`

예:
```markdown
## 라이선스별 기능 범위
| 기능 | 법무법인용 | 기업법무팀용 |
|-----|---------|-----------|
| 상담 등록 | ✓ | ✓ |
| 상담 분석 | ? _(확인 필요)_ | ? _(확인 필요)_ |
```

## 3구역 확인 추적 - 실제 적용

화면의 각 요소를 3구역으로 분류해 기록한다:

### 예시 1: 상태값 정의
```markdown
### 상태값
- 신규 (NEW) - 설계서 텍스트 명시 [✓] / 화면이미지 확인 [_]
- 진행중 (IN_PROGRESS) - 설계서 텍스트 명시 [✓] / 화면이미지 확인 [✓✓]
- 완료 (COMPLETED) - 설계서 텍스트 명시 [_] / 화면이미지 확인 [✓✓]
```

### 예시 2: UI 요소 위치
```markdown
### 검색 영역 배치
- 상담번호 입력: 텍스트만 [✓] / 이미지 확인 [✓✓]
- 상담 상태 필터: 텍스트만 [✓] / 이미지 확인 [✓✓]
- 숨김 항목 제어: 텍스트만 [_] / 이미지 확인 [✓] → disabled 확인 필요
```

## 화면설계서 3구역 분류 기준

### 구역 1: 텍스트만 확인 가능한 부분
- 필드명과 라벨
- 필드 속성 (필수, 길이, 타입)
- 버튼 기능 설명
- 상태값 정의 테이블
- 라이선스 구분 표기

**기록**: `_(텍스트 기반)_`

### 구역 2: 화면이미지에서 시각적으로 확인 필요한 부분
- UI 요소의 실제 위치와 크기
- 조건부 표시/숨김 상태
- disabled/readonly 상태
- 컬러와 아이콘
- 레이아웃과 배치
- 상태 변화 시 UI 변화

**기록**: `_(이미지 확인)_`

### 구역 3: 미확인
- 텍스트에도 없고, 이미지에도 보이지 않는 항목
- 추론이나 가정으로만 알 수 있는 항목

**기록**: `_(확인 필요)_`

## 체크리스트 - 추출 완료 여부 확인

분석을 마친 후 아래를 확인한다:

- [ ] 모듈명, GNB/LNB 위치 명확히 했는가?
- [ ] 기능 목적을 1-2문장으로 정의했는가?
- [ ] 화면 구성 (검색/목록/상세) 을 명시했는가?
- [ ] 모든 필드를 필수 여부와 함께 나열했는가?
- [ ] 버튼 동작과 연결 흐름을 기록했는가?
- [ ] 상태값/코드값을 전부 추출했는가?
- [ ] 라이선스 구분을 표로 정리했는가?
- [ ] 불확실한 항목을 `_(확인 필요)_` 표기했는가?
- [ ] 3구역 추적을 기록했는가?
- [ ] 설계 변경 이력을 history/requests/ 에 기록했는가?
