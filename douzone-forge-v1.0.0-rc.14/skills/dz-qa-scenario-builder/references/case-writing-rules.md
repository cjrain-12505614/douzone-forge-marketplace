# 테스트 케이스 작성 규칙

## 목표 밀도

실무자가 직접 작성한 가온 검수시나리오 수준의 밀도를 목표로 한다.
기능을 크게 묶은 요약형 시나리오가 아니라, 실제 검수자가 체크박스처럼 쓸 수 있는 상세 시나리오를 작성한다.

## 기본 원칙

1. `테스트 케이스 = 기능명 또는 검수 묶음명`
2. `1행 = 1행동 + 1예상결과`
3. `1기능 = 여러 행`
4. 화면 흐름 순서대로 작성
5. 정상, 권한, 기본값, 검증, 예외, 연계를 모두 포함
6. 버튼/필드/팝업/탭/상태/문구 확인을 각각 분리
7. 입력/선택 절차는 예시 값 또는 선택 항목까지 구체적으로 적는다.

## 컬럼별 작성 기준

- 메뉴명:
  - 실제 검수 대상 LNB
  - 예: `송무업무`
- 테스트 케이스:
  - 기능명 또는 검수 묶음명
  - 예: `사건일지 공통 추가`
- 1depth~3depth:
  - 기능이 속한 화면 경로
  - 예: `송무업무 / 사건일지 탭 / 사건일지 추가 팝업`
- 절차:
  - 실제 사용자 액션
  - 예: `[고객명]에 '테스트고객'을 입력한다.`
- 예상결과:
  - 해당 액션 후 나타나야 할 결과
  - 예: `고객명에 '테스트고객'이 입력된다.`

## 반드시 쪼개야 하는 경우

- 클릭 후 화면 변화가 있는 경우
- 팝업이 열리는 경우
- 기본값이나 초기 상태를 확인하는 경우
- 드롭다운/체크박스/라디오 선택지가 있는 경우
- 저장 전후 결과가 달라지는 경우
- 필수값, 길이 제한, 중복 체크, 계산, 상태값 검증이 있는 경우
- 다른 메뉴/탭/팝업으로 이어지는 경우
- 권한별 결과가 다른 경우

## 기본 체크 축

복잡한 화면은 아래 축을 순서대로 훑으며 작성한다.

- 메뉴 노출
- 메뉴 위치
- 메뉴 접근 권한
- 화면 기본 구성
- 검색영역 존재/기본값/선택지/초기화/조회
- 탭/리스트/카드/컬럼 노출
- 목록 선택과 상세영역 연동
- 버튼 노출/활성화/비활성 조건
- 팝업 호출
- 입력 필드 존재/기본값/형식/선택지
- 필수값 검증
- 저장/수정/삭제
- 상태 변경
- 후속 메뉴 연계
- 빈 데이터/권한 없음/예외 처리

## 좋은 분해 예시

나쁜 예:

- `사건일지 추가 팝업`

좋은 예:

- 테스트 케이스: `사건일지 공통 추가`
- 1depth~3depth: `송무업무 / 사건일지 탭 / 사건일지 추가 팝업`
- `[추가] 버튼을 클릭한다.`
- 사건일지 추가 팝업이 노출되는지 확인한다.
- 고객선택 팝업에서 `공통` 항목을 선택한다.
- 내용을 입력하고 저장한다.
- 다른 고객 타임라인에서도 동일 일지가 노출되는지 확인한다.

같은 기능 묶음의 연속 행은 `테스트 케이스`, `1depth~3depth` 셀 병합을 우선 검토한다.

입력/선택 예시:

- 절차: `[정렬]에서 '등록일순'을 선택한다.`
- 예상결과: `정렬값이 '등록일순'으로 변경되고 타임라인이 등록일 기준으로 재정렬된다.`
- 절차: `[고객명]에 '미등록고객_테스트'를 입력한다.`
- 예상결과: `고객명에 '미등록고객_테스트'가 입력되며 고객담당자 입력영역은 비활성화된다.`

예외 케이스는 별도 테스트 케이스로 분리한다.

## 권한과 분기 처리

같은 화면이라도 아래가 다르면 별도 테스트 케이스로 쓴다.

- 전체 권한 / 회사 권한 / 부서 권한
- 내부 진행 / 외부 진행
- 송무 / 자문 / 상담
- 요청 / 진행 / 완료 / 취소
- 직접 입력 / 팝업 선택 / 자동 계산

## 밀도 판단 기준

아래 중 하나라도 해당하면 케이스가 아직 얕을 가능성이 높다.

- 검색조건이 3개 이상인데 조회 케이스가 1~2건뿐인 경우
- 입력 팝업인데 `열기`와 `저장`만 있는 경우
- 상태값이 여러 개인데 상태 전환 케이스가 없는 경우
- 권한 차이가 있는데 권한별 케이스가 없는 경우
- 설계서 페이지 수에 비해 테스트 케이스 수가 지나치게 적은 경우

## 작성 완료 후 자체 점검

시나리오를 다 쓴 뒤에는 아래 질문으로 한 번 더 점검한다.

- 설계서 description에 나온 모든 상황을 절차로 분리했는가?
- 각 action마다 예외 케이스를 확인하는 절차가 필요한가?
- 저장 후 반영, 메시지, 권한, 빈 화면, 연계 영향까지 확인 절차가 있는가?
- 검색/필터에 결과 없음, 최소 선택 제약, 대표 조합 조회가 포함됐는가?
- 입력 항목에 최대길이, 허용문자, 공백, 비정상 입력 같은 validation 체크가 포함됐는가?
- 입력/선택 절차에 예시 값 또는 선택 항목이 적혀 있는가?
- 예외 케이스가 별도 테스트 케이스로 분리돼 있는가?
- 기본값, 초기 상태, 활성/비활성 조건을 확인하는 절차가 있는가?
- 보이면 안 되는 영역이나 수정되면 안 되는 상태까지 확인했는가?
- 같은 기능의 이전/후속 설계서, 공통 정책, 히스토리 문서를 대조해 추가 규칙을 반영했는가?
- 연관 문서와 주 설계서가 다르거나 우선순위가 불명확한 항목을 메모로 남겼는가?
- 재점검 중 새로 추가한 케이스를 하단에 방치하지 않고, 실제 검수 흐름에 맞게 다시 배치했는가?

이 질문에 `아니오`가 나오면 테스트 케이스를 추가하거나 기존 묶음 아래 절차를 보강한다.

## 재배치 원칙

- 추가 보강 케이스는 작성 편의를 위해 하단에 임시 작성할 수 있지만, 완료 전에는 반드시 본문 흐름으로 이동한다.
- 재배치 기준은 실제 검수자가 따라갈 순서다.
- 기본 흐름 예:
  - 탭/화면 진입
  - 기본 노출/초기 상태
  - 조회/검색/필터
  - 팝업 호출/입력/선택
  - 저장 후 반영
  - 연계 이력/후속 영향
  - 권한/예외
- 같은 기능 묶음의 예외 케이스는 해당 정상 케이스 바로 뒤에 둔다.
- 새 케이스를 끼워 넣은 뒤에는 순번, 셀 병합, 강조 서식이 흐름에 맞게 유지되는지 다시 확인한다.

## 설계 미흡 메모 규칙

- 설계서에 규칙이 없거나, 모호하거나, 추가 확인이 필요한 항목은 `예상결과`에 메모를 덧붙인다.
- 메모 형식 예:
  - `(화면설계 미기재: 주민번호 validation 규칙 확인 필요)`
  - `(화면설계 재확인 필요: 권한명 상세 정의 없음)`
- 시트 도구가 허용하면 메모가 포함된 `예상결과` 셀은 붉은색으로 강조한다.
- 메모는 케이스를 제거하는 용도가 아니라, 검수자에게 `설계 보강 필요 포인트`를 함께 전달하는 용도로 사용한다.
