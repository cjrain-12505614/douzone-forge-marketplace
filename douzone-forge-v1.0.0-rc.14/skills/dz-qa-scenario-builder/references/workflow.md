# 검수시나리오 생성 워크플로

## 1. 입력으로 받아야 하는 것

- 화면설계서 또는 설계서 경로
- 대상 구글시트 URL
- 요청자 이름

가능하면 함께 확인할 것:

- 메뉴 URL 또는 화면 코드
- 테스트용 URL/계정 정보
- 특별한 사전조건

## 2. 템플릿 시트 복제

기본 템플릿 자산:

- 템플릿 스프레드시트 ID: sheets-config.json의 `templates.qa-scenario.spreadsheet_id`
- 템플릿 탭: sheets-config.json의 `templates.qa-scenario.sheet_name`

절차:

1. 템플릿 시트 메타를 읽어 현재 탭명을 확인한다.
2. MCP 도구로 템플릿 탭을 대상 스프레드시트로 복제한다:
   ```
   MCP 도구: copy_sheet(
     spreadsheet_id="{템플릿ID}",
     source_sheet_name="{템플릿탭명}",
     destination_spreadsheet_id="{대상ID}"
   )
   ```
3. 복제된 새 탭명을 `[검수] {메뉴명 또는 기능명}`으로 변경한다:
   ```
   MCP 도구: rename_sheet(
     spreadsheet_id="{대상ID}",
     old_name="Copy of {템플릿탭명}",
     new_name="[검수] {메뉴명}"
   )
   ```

## 3. 헤더 입력 규칙

현재 템플릿 기준 기본 입력 위치:

- 통합테스트 ID: `F2`
- 작성자: `F3`
- 메뉴: `F4`
- 테스트 사전조건: `F5`
- 테스트환경: `F6`
- URL: `F7`
- 상세정보: `F8`

MCP 도구로 입력:
```
MCP 도구: update_cells(
  spreadsheet_id="{대상ID}",
  range="'{시트명}'!F2:F8",
  values=[
    ["TC-{LNB코드}-001"],
    ["SBUnit {요청자명}"],
    ["{메뉴명}"],
    ["{사전조건}"],
    ["크롬(Chrome)"],
    ["{URL}"],
    ["{상세정보}"]
  ]
)
```

입력값 규칙:

- 통합테스트 ID:
  - 형식 `TC-{LNB코드}-{순번3자리}`
  - 상단 기본값은 첫 케이스 기준 `TC-{LNB코드}-001`
- 작성자:
  - 기본값 `SBUnit {요청자명}`
- 메뉴:
  - 기획서 기준 메뉴명 또는 기능명
- 테스트 사전조건:
  - 기본값 유지
  - 특별한 전제조건이 있으면 그 값으로 교체
- 테스트환경:
  - 기본값 `크롬(Chrome)`
- URL:
  - 별도 URL이 없으면 템플릿 기본값 유지
  - 별도 URL이 있으면 교체
  - 테스트 사이트 계정 정보는 사용자 지시가 있을 때만 추가
- 상세정보:
  - 요청한 화면설계서명

## 4. LNB 코드 추출 규칙

메뉴 URL이 `https://.../#/모듈/GNB/LNB` 구조면 마지막 세그먼트를 LNB 코드로 본다.

예:

- `https://gwa.douzone.com/#/UR/URB1100/URB1100`
- LNB 코드: `URB1100`

URL이 없으면 화면설계서, 메뉴 코드 표기, 기존 컨텍스트에서 LNB 코드를 찾는다.
그래도 못 찾으면 사용자에게 확인이 필요하다.

## 5. 테스트 케이스 번호 규칙

본문의 테스트 케이스 번호는 단순 순번을 기본값으로 쓴다.

- `1`
- `2`
- `3`

검수시트 본문은 메뉴 코드형 번호를 쓰지 않고, 위에서 아래로 `1, 2, 3...` 순차 채번한다.

## 6. 테스트 케이스 묶음 규칙

- `테스트 케이스`는 화면명보다 기능명 또는 검수 묶음명으로 작성한다.
- 한 테스트 케이스 아래에 여러 `절차 / 예상결과` 행을 둘 수 있다.
- `1depth~3depth`는 해당 기능이 수행되는 화면 경로를 적는다.
- 값 입력이나 선택이 필요한 절차는 예시 값 또는 선택 항목을 함께 적는다.
- 예외 케이스가 있으면 다른 값, 다른 선택 항목, 비정상 입력 기준의 별도 테스트 케이스를 만든다.

예:

- 테스트 케이스: `사건일지 공통 추가`
- 1depth: `송무업무`
- 2depth: `사건일지 탭`
- 3depth: `사건일지 추가 팝업`

같은 테스트 케이스에 속한 연속 행은 `테스트 케이스`, `1depth~3depth` 셀 병합을 우선 검토한다.

## 7. 작성 완료 후 재점검 체크리스트

시나리오 초안 작성이 끝나면 아래 기준으로 반드시 한 번 더 점검한다.

1. 설계서의 description, 주석, 예시, 예외 설명에 나온 모든 상황이 절차로 분리됐는지 확인한다.
2. 각 action에 대해 예외 케이스가 발생할 수 있으면 확인 절차가 추가됐는지 본다.
3. 추가 확인이 필요한 저장 후 반영, 다른 탭/메뉴 연계, 메시지, 권한, 빈 화면 절차가 빠지지 않았는지 본다.
4. Validation이 필요한 입력칸은 형식, 최대길이, 최소길이, 허용문자, 공백, 범위, 비정상 입력까지 검토했는지 확인한다.
5. 입력/선택 절차에 예시 값 또는 선택 항목이 구체적으로 적혀 있는지 확인한다.
6. 정상 케이스와 예외 케이스가 한 테스트 케이스에 섞여 있지 않고 분리됐는지 확인한다.
7. 기본값, 초기 상태, 활성/비활성, 기본 선택값이 절차로 들어갔는지 확인한다.
8. 검색/필터는 정상 조회뿐 아니라 결과 없음, 최소 선택 제약, 대표 조합 조건을 포함했는지 확인한다.
9. 저장/수정/삭제/상태변경 기능은 처리 직후 화면 반영과 후속 영향까지 포함했는지 확인한다.
10. 권한/비노출 조건은 보여야 하는 경우뿐 아니라 보이면 안 되는 경우도 포함했는지 확인한다.
11. 같은 기능의 이전/후속 설계서, 공통 정책, 히스토리 문서를 함께 보고 주 설계서에 없는 공통 규칙이 있는지 확인한다.
12. 연관 문서와 주 설계서가 충돌하거나 우선순위가 불명확하면 `예상결과`에 재확인 메모를 남긴다.
13. 재점검 과정에서 추가된 테스트 케이스는 하단에 임시로 두지 말고, 실제 검수자가 따라갈 화면 흐름 순서에 맞게 본문 위치를 다시 정렬한다.

재점검 결과 누락이 있으면 기존 테스트 케이스 묶음에 절차를 추가하거나, 별도 기능 묶음을 새로 만든다.
새 기능 묶음을 추가한 경우에는 `탭 진입/기본화면 -> 기본 기능 -> 조회/필터 -> 입력/선택 -> 저장/반영 -> 연계/권한/예외` 같은 실제 검수 순서를 기준으로 적절한 위치에 끼워 넣는다.
설계서에 규칙이 없거나 불명확한 항목은 해당 행 `예상결과`에 `(화면설계 미기재: ...)`, `(화면설계 재확인 필요: ...)` 같은 메모를 추가한다.

## 8. 결과 영역 규칙

- 개발/AI/설계 검수율은 템플릿 수식을 유지한다.
- 조건부 서식도 템플릿 값을 유지한다.
- 새 시트 생성 시 요약 영역은 직접 다시 계산하지 않는다.

## 9. 민감정보 표시 규칙

사용자가 요청하면 다음 방식이 가능하다.

- URL과 테스트 계정은 그대로 표시
- 비밀번호는 표시만 마스킹
- 원문은 시트 내부 보조 셀에 유지

원문을 유지하지 말아야 하는 경우에는 시트 값 자체를 마스킹 값으로 덮어쓴다.
