# Amaranth 출력 매핑 규칙

## `context/`로 보내는 내용

- 화면 구성
- 검색조건, 컬럼, 버튼, 상태값
- 업무 규칙
- 메뉴별 라이선스 차이

예:
- 검수 시나리오에서 메뉴 구성과 저장 규칙이 명확히 드러난 경우
- 요구사항 시트에서 신규 메뉴의 구조가 반복 확인되는 경우

## `history/requests/`로 보내는 내용

- 고객사 요구 배경
- 왜 이런 기능이 생겼는지에 대한 이유
- 정책 변경 배경
- 차수/고객/운영 이력과 연결되는 요구사항

예:
- 택스로 대응으로 수익현황 집계 기준이 바뀐 경우
- 그룹사 우선 적용으로 기업법무팀 대응 범위가 재정의된 경우

## `history/reports/`로 보내는 내용

- 탭 전체 요약
- 검수율 집계
- WBS 진행 현황 요약
- 운영/AI/고객사 대응 흐름 정리
- backlog 요약

예:
- 고객사 대응 프로젝트 이력
- 검수 시나리오 미완료 항목 정리
- 월간 WBS 요약

## `tasks/_current.md`로 보내는 내용

- 지금 실제로 움직여야 하는 일
- 목표일이 있는 진행형 작업
- 블로커
- 오늘 확인할 일

예:
- 가온 대응 `79%`, 목표일 `2026-04-08`
- 그룹사 대응 `2026-02-05 기준 22%`, 최신 상태 확인 필요

## `tasks/enhancements/*.md`로 보내는 내용

- 하나의 고객사 대응
- 하나의 신규 메뉴
- 하나의 검수 후속 작업 묶음
- backlog에서 분리된 독립 개선 건

예:
- `gaon-requirements-response.md`
- `group-legal-support-response.md`
- `numbering-policy-followup.md`

## 작성 시 주의

- 진행형 자료는 기준 날짜를 같이 적는다.
- 표 전체를 복사하지 말고 재사용 가능한 구조로 바꾼다.
- `검수불가`는 원인까지 쓰지 못하면 최소한 접근 제약인지 기능 미구현인지라도 남긴다.
- 내부 선적용인지 외부 고객 대응인지 문맥을 명확히 적는다.
