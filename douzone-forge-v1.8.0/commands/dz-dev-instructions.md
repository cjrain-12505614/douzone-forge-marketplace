Claude Code에서 개발할 사항을 확인하는 커맨드다. 아래 순서대로 수행하라.

> **개발 지시서 출처 자동 판별 (H4 파라미터화)**: 프로젝트 구조에 따라 "구현 지시서"의 출처가 다르다. 고정 `docs/02_기획/요구사항/sprint{N}_*` 경로를 가정하지 말 것.
> - **A10 모듈 개발**(기본): 지시서 = forge 개발 파이프라인 산출물 — ③ 개발계획서(D-06, `규칙/프로세스/개발-계획-수립-표준.md` · 산출물은 `Amaranth10/{모듈}/history/plans/`) + ④ 개발설계 산출물(D-08, `규칙/프로세스/개발설계-산출물-작성-표준.md` — 로직·테이블·쿼리·API 정의서) + 관련 PRJ(`프로젝트/`) 02 TASK 목록·06_의사결정. 화면설계서(D-07)·요구사항이 상위 입력. (경로는 forge — 브리지 세션 `_forge/` 경유)
> - **forge 자체 PM 프로젝트**: 스프린트 구조면 기존 `docs/02_기획/요구사항/sprint{N}_*` 사용.

1. 작업 대상이 **A10 모듈**인지 **forge PM 프로젝트**인지 먼저 식별한다(브리지 켠 소스 레포면 A10).
2. **A10 모듈**: 해당 모듈의 D-06 개발계획서 + D-08 개발설계 산출물 + PRJ TASK 목록을 읽어 다음 개발 항목(Task)을 파악한다. **forge PM**: 활성 스프린트 지시서를 읽는다.
3. 내용을 기반으로 아래 형식으로 보고한다(스프린트 구조가 아니면 Step↔Task 로 치환):

```
## 현재 스프린트: Sprint {N}

### 전체 Step 현황
| Step | 항목 | SP | 상태 |
|------|------|-----|------|
| 0 | ... | - | ✅/⬜ |
| 1 | ... | 5 | ✅/⬜ |
| ... | | | |

### 다음 실행할 Step
**Step {X}: {제목}** ({SP} SP)

{지시서에서 해당 Step의 지시 내용 요약}

### 시작하려면
"Step {X} 진행해" 라고 말씀해주세요.
```

4. 이미 완료된 항목이 있는지 확인하려면:
   - **A10 모듈**: forge `프로젝트/PRJ-NNNN_*/03_개발/` 구현 보고와 모듈 `history/_timeline.md`, 그리고 소스 레포 4단계 브랜치(develop→devqa→sqa→master) 도달 지점을 확인한다(`docs/구현보고서/`를 가정하지 않는다).
   - **forge PM**: `docs/구현보고서/` 폴더의 보고서 존재를 확인한다.
   - Git 로그에서 관련 커밋(`[PRJ-.../TASK-...]` 프리픽스)이 있는지 확인한다.
   - 완료 여부가 불확실하면 해당 레포에서 빌드(`./gradlew clean build` — build.gradle 위치에서)를 실행하여 코드 상태를 점검한다.
