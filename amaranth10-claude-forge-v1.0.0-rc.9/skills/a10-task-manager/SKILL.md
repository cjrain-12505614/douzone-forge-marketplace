# 업무 관리자 (Task Manager)

Amaranth 10 모듈의 고도화 업무, 일정, 담당자를 관리하는 방법론.
현재 어떤 업무가 진행 중이고 무엇이 남아있는지를 Claude가 파악해
기획·개발 작업 시 우선순위와 일정 맥락을 자동으로 반영할 수 있게 한다.

## 업무 파일 계층 구조

```
{모듈명}/
└── tasks/
    ├── _current.md               # 현재 진행 중인 업무 목록 (항상 참고)
    └── enhancements/             # 고도화 건별 상세
        ├── {날짜}-{제목}.md
        └── ...
```

## 각 파일의 역할

### _current.md
지금 이 시점에서 진행 중이거나 예정된 업무의 전체 목록. 가장 자주 참고하는 파일.
- 진행 중인 고도화 건 목록 (상태, 일정, 담당자)
- 이번 스프린트/마일스톤 목표
- 블로커 또는 의존성 있는 항목
- 최근 완료된 건 (최근 3개월)

### enhancements/{날짜}-{제목}.md
개별 고도화 건의 상세 내용.
- 요청 배경 및 목적
- 요구사항 상세
- 영향 범위 (GNB/LNB, 라이선스)
- 일정 (요청일, 착수일, 목표 완료일, 실제 완료일)
- 담당자 (기획, 개발, QA)
- 현재 진행 상태 및 이슈

## 상태 값 정의

| 상태 | 의미 |
|-----|------|
| 검토중 | 요구사항 분석 및 가능 여부 검토 단계 |
| 기획중 | 스펙 작성 및 화면 설계 단계 |
| 개발중 | 구현 진행 중 |
| QA | 테스트 진행 중 |
| 완료 | 배포 완료 |
| 보류 | 일정 미정 또는 우선순위 하향 |

## 세션 재사용 원칙 (Session Reuse Principle)

같은 주제의 세션이 이미 진행 중이면 세션을 재개하고, 새 세션을 중복 생성하지 않는다.

예시:
```
1. tasks/_current.md와 sessions/_current.md 비교
2. 진행 중: "상담관리 고도화 건 분석" 세션
3. 이미 완료: 상담 조회, 상담 등록 요구사항 분석
4. 남은 것: 상담 수정, 상담 삭제 요구사항
→ 새 세션이 아니라 진행 중인 세션을 재개
```

## 작업 시작/종료 체크 (Work Session Markers)

에이전트 작업을 추적하기 위해 아래를 기록한다:

**작업 시작 시**:
```
작업 시작 체크: 2026-03-20 14:30
대상 업무: [업무명]
계획 단계: [현재 단계]
```

**작업 종료/중단 시**:
```
작업 종료 체크: 2026-03-20 16:45
완료: [완료한 항목]
남은 것: [다음 할 일]
중단 이유 (있으면): [사유]
```

## 로드 원칙

**우선 확인 파일:** `_개인/sessions/{모듈}/_current.md` 확인 후 `_current.md` 로드
**작업 시작 시:** `_current.md`로 현재 업무 현황 파악
**특정 고도화 건 작업 시:** 해당 `enhancements/` 파일 추가 로드
**기획 작업 시:** task-manager + context-manager + history-manager를 함께 로드하면
  현재 업무 우선순위, 기능 현황, 개발 배경을 모두 파악한 상태에서 기획 가능

## 출력 순서 표준 (Output Order Standard)

업무 현황을 요약할 때 아래 순서로 정리한다:

1. **진행 중** — 지금 작업 중인 업무 (상태: 기획중, 개발중, QA)
2. **예정** — 다음에 할 업무 (상태: 검토중, 계획 수립 필요)
3. **블로커** — 진행을 막는 것들 (의존성, 확인 대기 등)
4. **최근 완료** — 최근 3개월 완료된 업무

## 충돌 감지 (Conflict Detection)

현재 논의 중인 기능과 직접 충돌하는 건을 감지하면 별도로 강조한다:

예시:
```
⚠️ 충돌 주의
- 진행 중: "상담 상태 필드 추가"
- 예정: "상담 기본 정보 재설계"
→ 두 작업이 같은 화면을 다루고 있음. 우선순위 또는 일정 조정 필요
```

## 업무 추가 및 업데이트

`/add-task` 커맨드로 새 고도화 건을 등록하고,
`/load-tasks` 커맨드로 현재 업무 목록을 로드한다.
작업 완료 후 `/update-context` 로 상태를 반영한다.

## Google Sheets 연동 (google-sheets MCP 연결 시)

`amaranth10-claude-forge/sheets-config.json`에 연동 시트가 정의되어 있다.
**google-sheets MCP가 연결된 경우** 로컬 파일 외에 시트 데이터도 참조해 더 정확한 현황 제공.
**모든 모듈에 공통 적용** — 모듈별 연결 시트는 config에서 자동 결정되며, 플러그인 수정 불필요.

### 업무 현황 조회 시 자동 보완

```
1. 로컬 tasks/_current.md 읽기 (기본)
2. google-sheets MCP 가용 확인
3. 가용하면 → sheets-config.json 로드
4. config["module_index"][모듈명] 으로 연관 시트 목록 조회
5. local_task_file이 있는 시트만 추가 조회 → 로컬 내용과 대조
```

### 조회 방법 (sheets-manager 스킬 참조)

```python
# 모듈에 연결된 시트 목록 조회 (config 기반, 하드코딩 없음)
sheet_keys = config["module_index"][모듈명]

# 각 시트 읽기 (타입별 패턴은 sheets-manager SKILL.md 참조)
for key in sheet_keys:
    cfg = config["sheets"][key]
    # cfg["type"], cfg["spreadsheet_id"], cfg["columns"] 활용
```

### 로컬 파일과 시트 연동

`sheets-config.json`의 `local_task_file` 필드가 연동 파일을 지정.
`null`이면 읽기 전용(조회만). 새 모듈 추가 시 config에 항목만 추가하면 됨.

시트 → 로컬 동기화: `/sync-sheets [모듈명]`
시트 직접 조회: `/load-from-sheet [시트키 또는 모듈명]`


## 마크다운 링크 규칙 (필수)

마크다운 파일 작성·업데이트 시 **모든 경로·파일 참조는 클릭 가능한 상대 링크**로 작성한다.

```markdown
# 나쁜 예 (클릭 불가)
법무관리/tasks/_current.md에서 확인

# 좋은 예 (클릭 가능)
[법무관리/tasks/_current.md](../법무관리/tasks/_current.md)에서 확인
```

- 상대 경로는 현재 파일 위치 기준
- 코드블록 안의 폴더 구조 다이어그램은 예외 (링크 불필요)
- 상세 규칙은 douzone-forge CLAUDE.md의 "마크다운 링크 표기 규칙" 섹션 참조
